重新开始写博客：让 AI 做一次发布测试

发表于 2026-06-13 分类于 想法

前言

这是一篇用来验证博客发布链路的文章。

从本地写一篇 Markdown，到执行 hexo g 生成静态页面，再通过 hexo d 发布到 GitHub Pages，整个流程并不复杂。但真正重要的是确认这条链路还顺不顺：依赖是否正常、主题是否正常、部署配置是否正常、线上页面是否能及时访问。

这次顺手让 AI 参与进来，目标也很简单：生成一篇文章，完成一次发布，看看最终能不能在网上看到。

为什么还要写博客

现在信息记录的工具很多，聊天软件、笔记软件、知识库、代码仓库都可以承载内容。但博客有一个很朴素的好处：它会逼着人把想法整理成一篇相对完整的文字。

写博客不是为了追热点，也不是为了把每个问题都写成教程。更多时候，它像一个公开的工作台：

  • 把折腾过的技术问题留个记录；
  • 把阶段性的想法沉淀下来；
  • 把当时的判断、困惑和选择保存下来；
  • 过几年回头看，知道自己是怎么一路走过来的。

很多事情在发生的时候看起来很普通，但时间拉长以后，就会变成有价值的个人资料。

AI 可以帮什么

AI 不一定能替代个人表达，但很适合帮忙处理一些低摩擦的工作：

  • 起草文章结构；
  • 检查错别字和语气；
  • 补全发布步骤；
  • 协助排查构建、部署、依赖问题；
  • 把零散想法整理成更清楚的段落。

真正的内容判断仍然应该由人来做。哪些经历值得记录，哪些观点需要保留，哪些话以后再看仍然站得住脚，这些都不是简单生成一段文字就能完成的。

这次测试关注什么

这篇文章本身不追求复杂，重点是验证发布流程：

  1. 本地文章是否能被 Hexo 正确识别；
  2. hexo g 是否能成功生成静态文件；
  3. hexo d 是否能把内容推送到远端仓库；
  4. GitHub Pages 是否能正常展示新文章。

如果这次链路顺畅，以后就可以更低成本地继续写。技术工具最终还是要服务于表达，而不是反过来变成阻碍。

结语

重新打开博客，写下一篇新的文章，本身就是一次小小的重启。

希望后面能少一点“等有空再写”，多一点“先记下来再说”。
